## 一、Hyperliquid 清算機制

### 清算原理

```
你開倉時：
  保證金 = 名目價值 / 槓桿
  例如：$712 / 10x = $71.2 保證金

當價格反向移動時：
  名目價值縮水 → 保證金比率下降
  保證金比率 < 維持保證金率 → 清算觸發
  
多頭清算價 ≈ 入場價 × (1 - 0.8/槓桿)
空頭清算價 ≈ 入場價 × (1 + 0.8/槓桿)
```

### 各幣種維持保證金率

| 幣種 | ≤4x | 5-10x | 11-20x | >20x |
|------|-----|-------|--------|------|
| BTC/ETH | 4% | 4% | 5% | 6% |
| SOL/UNI/HYPE/AAVE | 4% | 5% | 6% | 8% |

## 二、風險管理 SOP（標準作業流程）

### A. 每日持倉風險檢查

```bash
python3 ~/.hermes/scripts/hl_risk_monitor.py
```

**檢查項目：**
```
□ 保證金使用率 < 70%？
□ 所有持倉距離清算 > 15%？
□ 單筆最大虧損 < 5%？
□ 有掛止損單（reduceOnly）？
□ 資金費率正常（非極端）？

如果任何一項否：
→ 立即減少倉位或追加保證金
```

### B. 入場前的倉位計算

```bash
# 計算安全倉位
python3 ~/.hermes/scripts/position_calc.py \
  --account 66 --risk 2 --lev 5 --entry XX --coin XXX
```

**入場前必須確認：**
```
□ 單筆風險 ≤ 2% 帳戶？
□ R:R ≥ 1:2？
□ 止損距離清算 > 5%？
□ 槓桿 ≤ 5x？
□ 總保證金使用率 < 60%？
□ 與現有持倉相關性 < 0.7？
```

### C. 止損後的恢復流程

```
Step 1: 確認止損已執行
Step 2: 記錄虧損金額和原因
Step 3: 檢查剩餘倉位風險
Step 4: 虧損 ≤ 3% 帳戶 → 休息 1 小時
        虧損 > 3% 帳戶 → 休息 1 天
Step 5: 虧損 > 5% 帳戶 → 全面停止，重新檢視策略
Step 6: 寫交易日誌
Step 7: 等心態恢復後再考慮新倉位
```

### D. 重大事件前的風險降低

```
FOMC / CPI / NFP 前 2 小時：
□ 減少倉位 30-50%
□ 確認所有止損單已掛好
□ 不開新倉位

事件發生後等待 30 分鐘再操作
```

## 五、緊急情況處理

### 情境：持倉接近清算

```
信號：距清算 < 5%
Step 1: 不要再觀望
Step 2: 二選一：
  a) 追加保證金（增加帳戶價值）
  b) 部分平倉（減少名目價值）
Step 3: 如果來不及 → 全部平倉比被清算好
  （清算手續費通常 1-2%，大於手動平倉手續費）
```

### 情境：全市場暴跌

```
信號：BTC 1h 跌 > 10%
Step 1: 檢查所有持倉距清算距離
Step 2: 接近清算的 → 立即平倉
Step 3: 重倉的 → 減倉 50%
Step 4: 等待穩定後再重新評估
Step 5: 不要在恐慌中撈底
```

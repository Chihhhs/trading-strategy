# Log And Record Paths

## Purpose

This document explains where runtime records are written and which files an analyst should read for:

- trade-by-trade analysis
- Hyperliquid API debugging
- current live state inspection

The path rules below reflect the current implementation in:

- [src/trading_strategy/live/config.py](/D:/code/trading-strategy/src/trading_strategy/live/config.py:1)
- [src/trading_strategy/live/io.py](/D:/code/trading-strategy/src/trading_strategy/live/io.py:1)

## Directory Layout

### Live state

- Directory: [data/paper_strategies_live](/D:/code/trading-strategy/data/paper_strategies_live)
- Main file: [data/paper_strategies_live/live_state.json](/D:/code/trading-strategy/data/paper_strategies_live/live_state.json)

Use this when you want the latest local runtime view of:

- open positions
- managed orders
- protection status
- aggregate stats

### Trade history

- Directory: [data/trade_history](/D:/code/trading-strategy/data/trade_history)
- File pattern: `YYYY-MM-DD.jsonl`
- Example: [data/trade_history/2026-07-05.jsonl](/D:/code/trading-strategy/data/trade_history/2026-07-05.jsonl)

Use this as the primary source for post-trade or optimization analysis.

Each line is one JSON record written by `record_trade_event(...)`.

Typical events include:

- `position_opened`
- `position_close_submitted`
- `position_closed_reconciled`
- `entry_skipped`
- `entry_order_rejected`
- `run_summary`

For completed-trade analysis, also read the `history` array inside `live_state.json`, because closed trades are normalized there with strategy context such as:

- `entry_reason`
- `signal_reason`
- `signal_score`
- `btc_dir_at_entry`
- `risk_pct`
- `entry_order_type`
- `exit_reason`
- `exit_policy`
- `hold_minutes`
- `pnl`
- `pnl_pct`
- `outcome`

### Hyperliquid API debug log

- Directory: [data/hl_api](/D:/code/trading-strategy/data/hl_api)
- File pattern: `YYYY-MM-DD.log`
- Example: [data/hl_api/2026-07-05.log](/D:/code/trading-strategy/data/hl_api/2026-07-05.log)

This file is only written when `DEBUG_API` is enabled.

Use it for:

- raw Hyperliquid request/response troubleshooting
- order submission debugging
- trigger-price normalization checks
- verification mismatches between local state and exchange responses

This is not the recommended source for trade-performance analysis.

## Rotation Logic

Both log types are rotated by local calendar date.

- Trade history path is generated by `config.get_trade_log_path()`
- API debug path is generated by `config.get_api_log_path()`
- Current format uses `datetime.now().strftime("%Y-%m-%d")`

That means:

- all records from the same local day go into the same file
- a new file starts automatically after the date changes
- no manual cleanup is required for daily separation

## Analyst Guidance

### Use `trade_history` when:

- measuring entry quality
- analyzing skipped entries
- studying rejection reasons
- reviewing run-level blockers
- building optimization datasets from execution events

### Use `live_state.json` when:

- checking what positions are open right now
- verifying TP/SL attachment status
- inspecting reconciled exchange state
- reading normalized closed-trade summaries in `history`

### Use `hl_api` when:

- debugging exchange behavior
- inspecting raw API-side order outcomes
- validating price rounding or trigger normalization

## Notes

- `trade_history` and `hl_api` are intentionally separated so analysis data does not mix with low-level API diagnostics.
- `live_state.json` remains outside those directories because it is a mutable state snapshot, not an append-only analysis log.
